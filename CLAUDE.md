## Commands

### Run the application
```bash
./gradlew bootRun
# Or with explicit profile:
./gradlew bootRun -Pspring.profiles.active=dev
```

### Build (skip tests)
```bash
./gradlew clean build -x test
```

### Run tests
```bash
./gradlew test
# Run a single test class:
./gradlew test --tests "com.astraval.coreflow.SomeTest"
```

### Production deployment
```bash
./gradlew clean build -x test
sudo cp build/libs/coreflow-0.0.1-SNAPSHOT.jar /var/www/coreflow/coreflow.jar
sudo systemctl restart coreflow
```

## Architecture

### Stack
- **Java 21** + **Spring Boot 4.0.1**
- **PostgreSQL** via Spring Data JPA (Hibernate, `ddl-auto: update`)
- **JWT** authentication (jjwt 0.11.5) with access + refresh tokens
- **MapStruct** for DTO <-> entity mapping
- **Lombok** for boilerplate reduction
- **Apache Tika** for document/OCR text extraction
- **Logstash Logback encoder** for structured JSON logging

### Configuration profiles
- `application.yml` — shared config (JWT, mail, JPA, file upload paths)
- `application-dev.yml` — local DB (`localhost:5432/coreflow`, `postgres/1234567890`), port 8080
- `application-prod.yaml` — production overrides

The active profile defaults to `dev` when using `bootRun`. Override with `-Pspring.profiles.active=prod`.

### Security
`JwtAuthenticationFilter` (a `OncePerRequestFilter`) validates Bearer tokens on every request. It:
- Skips `/api/auth/**` and `/api/test/**` (public endpoints)
- Extracts `sub` (userId as `Long`) and `role` claim from the JWT
- Sets `ROLE_<ROLE>` authority in `SecurityContextHolder`
- Falls through silently on invalid/missing tokens (Spring Security rules enforce auth downstream)

`JpaConfig` provides `AuditorAware<Long>` that reads the authenticated userId from the security context to populate `created_by`/`modified_by` audit columns automatically.

### Module structure
All business logic lives under `src/main/java/com/astraval/coreflow/modules/`. Each module follows:
```
<module>/
  <Entity>.java          # JPA entity
  <Entity>Controller.java
  <Entity>Service.java
  <Entity>Repository.java
  <Entity>Mapper.java    # MapStruct interface
  dto/                   # Request/response DTOs
  projection/            # JPA projections (where used)
```

Key modules:
- **Auth** — login, register, refresh token (public endpoints, no JWT required)
- **user** — `User` entity, `UserService`, `UserRepository`
- **companies** — `Companies` entity; central multi-tenant anchor (items, orders, etc. all FK to a company)
- **customer** / **vendor** — `Customers`/`Vendors` entities; `CustomerVendorLink` tracks B2B relationships
- **items** — `Items` entity with `ItemType` (GOODS/SERVICE), `UnitType`, per-customer pricing (`ItemCustomerPrice`), per-vendor pricing (`ItemVendorPrice`), and stock tracking (`ItemStocks`)
- **orderdetails** — `OrderDetails` entity (`order_details` table); mutable live orders with `sellerCompany`, `buyerCompany`, `customers`, `vendors` FKs; split into `PurchaseOrderDetailsController` / `SalesOrderDetailsController`
- **ordersnapshot** — `OrderSnapshot` entity (`order_snapshot` table); immutable copy of an order at a point in time, linked back to `OrderDetails` via `orderReference`; mirrors orderdetails structure with separate Purchase/Sales controllers and services
- **orderitemdetails** — `OrderItemDetails`; line items attached to live `OrderDetails`
- **orderitemsnapshot** — `OrderItemSnapshot`; line items attached to `OrderSnapshot`
- **filestorage** — `FileStorage` entity; tracks uploaded files; upload paths configured under `app.file.upload.*`
- **address** — `Address` entity with JPA projection
- **emailtemplate** — `EmailTemplates` entity + `EmailTemplateService`; used for sending templated emails via Gmail SMTP
- **invitation** — DTOs only (no controller/service found at top level); likely consumed by other modules

### API response pattern
All controllers return `ApiResponse<T>` built via `ApiResponseFactory`:
- `ok(data, msg)` → 200
- `created(data, msg)` → 201
- `accepted(data, msg)` → 202
- `updated(data, msg)` → 203
- `deleted(msg)` → 204
- `validation(data, msg)` → 444 (custom code for field-level validation errors)
- `badRequest(msg)` → 400, `UnauthorizedAccess(msg)` → 401

Exception handling is split across `common/exception/handler/`: `GlobalExceptionHandler`, `AdminExceptionHandler`, `AuthExceptionHandler`, `DatabaseExceptionHandler`, `SystemExceptionHandler`.

### Order number / Payment number generation
- Order numbers: `ORD-MMYYYY-SEQ` — generated by the DB function `generate_order_number(companyId)`, backed by the `company_order_sequence` table.
- Payment numbers: `PAY-MMYYYY-SEQ` — generated by `generate_payment_number(companyId)`, backed by `company_payment_sequence`.
- Both sequences are per-company per-month and auto-increment atomically inside the DB function.
- Java side: `OrderDetailsService.getNextSequenceNumber(companyId)` / `PaymentService.getNextPaymentNumber(companyId)` call the respective repository native queries.

### Single-source table design for Orders and Payments

This is the most important architectural concept in the application. **A single row in `order_details` or `payments` is shared and visible to both companies in a B2B transaction.** There is no duplication of records between buyer and seller — both parties read from the same row.

#### Party model: Customers, Vendors, and the link

Every company manages its own contact lists:
- **`Customers`** — a company's list of buyers. Belongs to one `company` (the seller). Has an optional `customerCompany` FK pointing to a `Companies` record, set only if that customer is also a registered company on the platform (i.e., a linked party).
- **`Vendors`** — a company's list of suppliers. Belongs to one `company` (the buyer). Has an optional `vendorCompany` FK, set only if that vendor is also a registered company on the platform.
- **`CustomerVendorLink`** — when two companies are mutually linked (via invitation), one record in this table bridges the seller's `Customers` record to the buyer's `Vendors` record. Unique constraint on `customer_id` — one customer entry maps to at most one vendor entry.

**Linked vs. unlinked parties:**
- **Unlinked**: `customerCompany` / `vendorCompany` is `null`. Orders/payments created by that company are private — only that company can see them.
- **Linked**: `customerCompany` / `vendorCompany` is set. Orders/payments automatically populate both sides, making the record visible to both companies.

#### How Orders work (single-source)

`OrderDetails` columns that matter:
| Column | Meaning |
|---|---|
| `seller_company` | The selling company (order creator for sales) |
| `buyer_company` | The buying company (order creator for purchase) |
| `customer` | Seller's `Customers` record for the buyer |
| `vendor` | Buyer's `Vendors` record for the seller |

**Sales order creation** (`SalesOrderDetailsService.createSalesOrder`, `companyId` = seller):
1. Sets `sellerCompany` = caller's company.
2. Sets `customers` = the chosen customer record.
3. If `customer.customerCompany != null` (linked): sets `buyerCompany` = `customer.customerCompany`, then resolves the corresponding `Vendors` record on the buyer side via `VendorService.getBuyerVendorId` and sets `vendors`.
4. Sends a notification to the buyer company.

**Purchase order creation** (`PurchaseOrderDetailsService.createPurchaseOrder`, `companyId` = buyer):
1. Sets `buyerCompany` = caller's company.
2. Sets `vendors` = the chosen vendor record.
3. If `vendor.vendorCompany != null` (linked): sets `sellerCompany` = `vendor.vendorCompany`, then resolves the corresponding `Customers` record on the seller side via `CustomerService.getSellersCustomerId` and sets `customers`.
4. Sends a notification to the seller company.

**Reading orders — same row, different perspective:**
- `SalesOrderDetailsRepository.findOrdersByCompanyId(companyId)` — filters by `seller_company = companyId` (seller's view of their outgoing orders).
- `PurchaseOrderDetailsRepository.findPurchaseOrdersByCompanyId(companyId)` — filters by `buyer_company = companyId` (buyer's view of their incoming orders).

#### How Payments work (single-source)

`Payments` columns that matter:
| Column | Meaning |
|---|---|
| `receiver_comp` | Company receiving the money (seller) |
| `sender_comp` | Company sending the money (buyer) |
| `customer` | Receiver's `Customers` record for the payer |
| `vendor` | Sender's `Vendors` record for the payee |

**Buyer payment creation** (`BuyerPaymentService.createBuyerPayment`, `companyId` = buyer/sender):
1. Sets `senderComp` = caller's company, `vendors` = chosen vendor.
2. If `vendor.vendorCompany != null` (linked): sets `receiverComp` = `vendor.vendorCompany`, resolves the `CustomerVendorLink` to find the `Customers` record on the receiver side, sets `customers`.

**Seller payment creation** (`SellerPaymentService.createSellerPayment`, `companyId` = seller/receiver):
1. Sets `receiverComp` = caller's company, `customers` = chosen customer.
2. If `customer.customerCompany != null` (linked): sets `senderComp` = `customer.customerCompany`, resolves the `CustomerVendorLink` to find the `Vendors` record on the sender side, sets `vendors`.

**Reading payments — same row, different perspective:**
- `findPayerPaymentSummaryByCompanyIdNative(companyId)` — filters by `sender_comp = companyId` (buyer's view).
- `findPayeePaymentSummaryByCompanyIdNative(companyId)` — filters by `receiver_comp = companyId` (seller's view).

#### Due amount tracking

`Customers.dueAmount` and `Vendors.dueAmount` are **cached computed balances**, recalculated by `PartnerBalanceService` after every order or payment mutation.

- `fn_customer_due_amount(customerId)` = SUM of sales order totals for that customer − SUM of payments received (excludes quotation-status orders and declined payments).
- `fn_vendor_due_amount(vendorId)` = SUM of purchase order totals for that vendor − SUM of payments made.
- `PartnerBalanceService.refreshDueAmountsForOrder(order)` and `refreshDueAmountsForPayment(payment)` are called at the end of every create/update/delete operation.
- A materialized view `mv_party_due_summary` exists for reporting; refresh it with `SELECT refresh_mv_party_due_summary()`.

#### Rules for working with this design
- **Always populate both sides** when the counterparty is linked (`customerCompany`/`vendorCompany != null`). Failing to do so makes the record invisible to one party.
- **Always call `partnerBalanceService.refreshDueAmounts*`** after any mutation to orders or payments, or the cached due amounts will be stale.
- The `CustomerVendorLink` is the bridge for resolving the counterpart record — use it to find the `customer` from a `vendor` or vice versa, never assume by company ID alone.

### File uploads
Files are stored on the local filesystem. Paths:
- Item images: `uploads/item_img`
- Payment proofs: `uploads/payment_proof`

The `upload.path` root is set in `application-dev.yml`. OCR uses Tesseract CLI (`C:\Program Files\Tesseract-OCR\tesseract.exe` in dev).

### Audit fields
All entities use Spring Data JPA auditing (`@CreatedBy`, `@CreatedDate`, `@LastModifiedBy`, `@LastModifiedDate`) with `AuditingEntityListener`. The auditor is the authenticated user's ID (`Long`), defaulting to `0L` for anonymous operations.
